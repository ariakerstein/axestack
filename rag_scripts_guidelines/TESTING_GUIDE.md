# Automated Testing Guide

**Date**: October 29, 2025
**Purpose**: Automated testing for RAG pipeline improvements (semantic search + Claude 3.5)

## Quick Start

### 1. Run Baseline Tests (Before Changes)
```bash
cd rag_scripts_guidelines
./run_tests.sh
# Select option: 1 (baseline)
```

**What it does**:
- Runs 8 test questions through current system
- Captures latency, answer length, success rate
- Saves results to `test_results_baseline_YYYYMMDD_HHMMSS.json`

---

### 2. Deploy Changes
```bash
# Deploy SQL function (manual step - see DEPLOYMENT_SUMMARY.md)
# Then deploy edge function:
npx supabase functions deploy direct-navis --project-ref felofmlhqwcdpiyjgstx --no-verify-jwt
```

---

### 3. Run Post-Change Tests
```bash
./run_tests.sh
# Select option: 2 (post_change)
```

**What it does**:
- Runs SAME 8 questions through updated system
- Captures metrics with new semantic search + Claude 3.5
- Saves results to `test_results_post_change_YYYYMMDD_HHMMSS.json`

---

### 4. Compare Results
```bash
python3 compare_results.py
```

**What it does**:
- Loads latest baseline and post-change results
- Calculates latency improvements
- Shows question-by-question comparison
- Generates SQL for database quality metrics

## Database Quality Metrics

After running tests, you can fetch detailed quality scores from the database:

```sql
-- Generated by compare_results.py
SELECT
  b.question,
  b.accuracy_score as baseline_accuracy,
  p.accuracy_score as post_accuracy,
  ROUND(p.accuracy_score - b.accuracy_score, 2) as accuracy_delta,
  b.total_sec as baseline_sec,
  p.total_sec as post_sec,
  ROUND(p.total_sec - b.total_sec, 2) as sec_delta,
  b.num_chunks as baseline_chunks,
  p.num_chunks as post_chunks
FROM baseline b
FULL OUTER JOIN post_change p ON LOWER(TRIM(b.question)) = LOWER(TRIM(p.question));
```

Quality scores to compare:
- `accuracy_score`: How accurate is the medical information?
- `completeness_score`: Does it answer the full question?
- `source_support_score`: Are sources properly cited?
- `clarity_score`: Is it easy to understand?
- `overall_confidence`: Overall quality rating

## Success Criteria

After running tests and comparison, check:

- ✅ Latency reduced by 40-60%
- ✅ Success rate 100% (or same as baseline)
- ✅ Quality scores within ±0.05 of baseline
- ✅ Search method changed to `semantic_vector_similarity`
- ✅ Chunks retrieved consistently (5-8 per query)
- ✅ No errors in function logs

If all pass: **Deployment successful!** 🎉

If any fail:
- Check function logs for errors
- Review comparison output for patterns
- Consider adjusting similarity threshold (currently 0.65)
- May need to revert changes (see rollback plan in DEPLOYMENT_SUMMARY.md)
